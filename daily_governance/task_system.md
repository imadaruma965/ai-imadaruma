# 任務・期限の統治（タスクシステム）

> **やさしい要約（先に読む）**: `ONE_SYSTEM.md`  
> **いまほしいもの**: 手帳＋タスク帳を合体した **タスク管理と進捗**。携帯でもPCでも同じ一台。  
> **くわしい信用施策**: `ops_reliability.md`  
> **管轄**: 尊徳（日次）／栄一（仕組み）／君主（実行）

---

## 結論（先に）

| 器 | 役割 |
|----|------|
| **義務アプリ（最大1つ・可）** | 日付・納期・カテゴリ・一覧。外出先の主戦場 |
| **`ops_commitments.md`** | 国家OS側の要約／週次棚卸し（アプリ採用後は「写し」でも可） |
| **`today.md`** | **今日やる分だけ**（最重要3＋義務から引っ張った1〜2） |
| **`parking_lot.md`** | 日付のない思いつき・今やらぬ構想 |
| **`sales_pipeline.md`** | 営業案件（名前付き）。期限は義務アプリにも日期 |

**アプリはダメか？ → 条件付きで是である。**  
ダメなのは「探す・比較する・自作する・二つ持つ」。  
必要なのは **日付UIと一覧がある一台** で、Markdown表の限界を補うことである。

やらない戦の「AIツール開発」＝商品化・自作逃避。  
市販の一本を **選んで固定し、中身を埋める** のは内政である。

---

## なぜ後回しになるか（仕組み側）

1. **捕獲漏れ** — 請求書・口約束が「記憶」のまま  
2. **todayが願望で埋まる** — 義務が最重要3に入らない  
3. **期限が見えない** — 予定日の3日前に浮かび上がらない  
4. **支払い／発注が「タスク未満」** — 売上タスクの影に隠れる  

対策は、義務を **先に台帳へ**、朝に **期限順で today へ最大2件**。

---

## 運用（毎日の型）

### 捕まえた瞬間（30秒）

**PCの前**: `ops_commitments.md` に1行（締切・内容・種類）。  
**外出中・LINE**: `mobile_capture.md` に従い、スマホ INBOX へ1行 or 60秒返信。頭に置かない。

### 外出中（重要）

処理の本番はしない。**捕獲→定時窓（昼／夕）→夜に台帳へ**。詳細は `mobile_capture.md`。

### 朝（閣議・2分追加）

1. `ops_commitments.md` の **期限切れ／本日期限／3日以内** を見る  
2. その中から **今日やる義務を1〜2件** `today.md` に転記（売上3の枠より義務が先の日もあってよい）  
3. 主戦場タスクは残り時間で  

### 日中

`today.md` だけ見る。台帳は開かない（脱線防止）。

### 夜

完了した義務に ✅。繰り越しは台帳の日付を翌営業日にずらすか、事実を §8b に1行。

### 週1（金 or 日・10分）

台帳の棚卸し: 過ぎた・不用・曖昧な行を消すか日付を確定。曖昧なまま置かない。

## 種類（台帳の分類）

| 略 | 種類 | 例 |
|----|------|-----|
| **PAY** | 支払い | 請求・税金・サブスク・振込 |
| **PO** | 発注・手配 | 発注・手配・契約手続き |
| **SCH** | 予定 | 面談・収録・医療・会議 |
| **DL** | 納期・提出 | 納品・原稿・書類提出 |
| **REP** | 返信・確認義務 | 「金曜までに返す」系 |
| **ADM** | その他行政 | 役所・更新・手続き |

## 優先の理（尊徳）

朝、衝突したらこの順:

1. **期限切れ**（すでに遅い）  
2. **本日期限**  
3. **3日以内の PAY / PO / DL**（金と信用）  
4. 売上の最重要  
5. 発信・その他  

「計画して実践」は、大きな年次計画より **台帳→today の毎日の落とし込み** が本体である。大きな計画は栄一のロードマップのまま。

## 義務アプリ（条件付き許可）

今さんが欲しいもの——自動日付・納品日の選択・カテゴリ・全体一覧——は **Markdown表よりアプリが適する**。だから一本に限って採用してよい。

### 必須条件（これを満たすものだけ）

| # | 条件 |
|---|------|
| 1 | **スマホとPCの両方**で、同じリストが見られる |
| 2 | 期限・日付を選べる（納品予定など） |
| 3 | カテゴリ（リスト）で支払い・発注・予定・返信などが分かれる |
| 4 | **今日／期限順／全部**が一覧できる |
| 5 | 外出先でLINE直後に **15秒で1件追加**できる |
| 6 | **無料または既契約**で足りる |

### 採用の手続き（探検を防ぐ）

1. 下の候補から ** Tonight 中に一つ選ぶ**（比較表は作らない）  
2. `task_system.md` 末尾の「採用ロック」に名前を書く  
3. **14日間は乗り換え禁止**  
4. 既存の期限を全部そこへ移す（空アプリ禁止）  
5. 朝: アプリの「今日・期限切れ」→ `today.md` に義務1〜2  
6. 週1: アプリを見て、国家側に残したいものだけ `ops_commitments` か weekly に要点  

### 候補（余が絞ったもの・この中から）

| 候補 | 向く理由 | 向かぬ場合 |
|------|----------|------------|
| **Microsoft To Do** | **PC＋スマホ両方**。いまさんの条件にいちばん近い | 凝ったDBが欲しい人には物足りぬ |
| **TickTick** | 一覧・カレンダー強い。PCアプリあり | 機能が多くて設定に逃げるリスク |
| **Googleタスク＋カレンダー** | 予定と一体。Android／Gmail勢 | 一覧の弱さを感じる人も |
| **Appleリマインダー** | iPhoneだけなら最短 | **Windows PC中心だと弱い** |

**Notion／高機能DB**は「表が美しい」止まりやすい。今期の第一選択にしない。習熟済みなら例外。

### Markdown台帳との役割分け（アプリ採用後）

| 真実の置き場 | 内容 |
|--------------|------|
| **アプリ** | 生きている期限・納品予定・カテゴリ・一覧・リマインダ |
| **`ops_commitments.md`** | 週次の国家メモ、または「今月の重要義務」要約（全部写さなくてよい） |
| **`today.md`** | 当日の実行指示のみ |

アプリを入れても、**朝に today へ落とす型**と **mobile_capture の定時窓**は残す。道具が変わっても統治の型は同じである。

## 段階

| 段階 | 内容 | 期間目安 |
|------|------|----------|
| **0** | 頭の中の期限を洗い出す（メモでもアプリでも） | 今日〜明日 |
| **0.5** | 義務アプリを1つ選びロック。全件投入 | Tonight〜48時間 |
| **1** | 朝に義務1〜2を today へ。LINEは捕獲＋仮返答 | 1週間 |
| **2** | 週次棚卸し＋繰り越し2回は分割 | 定常 |
| **3** | 通知の例外ピン留め等（全体自動化はしない） | 習慣後 |

---

## やらないこと

- ToDoアプリの **探検・比較・乗り換え週**  
- アプリを2つ以上並行  
- 自作タスク管理AI／大Notion設計  
- today に全タスクを列挙（見なければ同じ）  
- 期限のない「いつかやりたい」を義務アプリに混ぜる → parking_lot
