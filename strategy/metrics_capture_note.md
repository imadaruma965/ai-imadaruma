# 数値・振り返り取得の方針

> **管轄**: 栄一（設計）／尊徳（週次記入の督促）／正篤（NL取込）  
> **正本の置き場**: `daily_governance/governance_board.md`  
> **日次の芽**: `night_review.md` §8b → `archives/daily_logs/`  
> **更新日**: 2026-07-14  

---

## 結論（先に）

**打ち方の手順書**: `daily_governance/HOW_TO_LOG.md`（迷ったら先にこっち）

1. **実践の日報化は是**である。毎日1〜3行残し、週末に拾ってNLの「今週の実践」にする。  
2. **フォロワー等を毎日打ち込むのは否**（過負荷）。週1のボードで足りる。  
3. **各SNSから自動取得は「技術的には一部可能、いまやる価値は薄い」**。専用連携を先に作るな。

---

## 構想どおりの配管（推奨）

```
[毎日・5分] night_review §8b
  実践1行＋失敗任意
  → archives/daily_logs/YYYY-MM-DD.md に保存（既存運用）

[金〜土・10〜15分] governance_board.md
  今週の daily_logs を辿る
  → ネタになりそうな実践を2〜3個ピック
  → 数字は週次で一括（目視転記）

[金〜日] 正篤
  ボードだけ読んで NL 起稿
```

これは「日報アプリ」ではない。**既存の夜レビューに短いNL用欄を足しただけ**である。

## 「ログを辿って実践文書にできるか」

| 問 | 答え |
|----|------|
| 可能か | **可能。むしろこれが正しい** |
| 誰がピックするか | 段階0: 君主がボードのA欄に転記。慣れたら正篤に「今週の daily_logs を読め、候補3つ出せ」 |
| 毎日長く書くか | **否**。長いと続かない。§8bは1〜3行 |

ガリア戦記は「その場の詳録」より、「週の戦いを三人称で再構成」でよい。日報は原料、NLは文である。

## 数字：毎日手打ち vs 自動連携

### 毎日手打ち

- 可能だが **分度を壊す**（離脱の元）  
- 体重・習慣は毎日でもよい（内政）。フォロワーは **週1**  

### 自動でフォロワーを取る——可能か

| 媒体 | 現実 |
|------|------|
| **YouTube** | Data API で登録者取得は比較的しやすい（鍵・割当あり） |
| **Instagram / Facebook** | Meta Graph。ビジネス／クリエイター連携が前提。手間あり |
| **X** | APIは有料プラン寄り。個人の趣味連携には重い |
| **LinkedIn** | 公式APIは厳しく、個人フォロワーの簡単取得は期待しにくい |
| **LINE** | 公式アカウント管理画面／Messaging API。連携設計が必要 |
| **ニュースレター購読者** | LI画面の目視が現実的なことが多い |

結論:

- **スクレイピングで無断取得はするな**（規約・壊れやすさ）  
- **「全部自動」は今期のやらない戦に近い**（ツール開発で売上から逃げる）  
- 段階2なら、苦痛な1媒体だけ公式APIか、市販の分析／予約ツールの数字を週1転記

## 段階の再掲

| 段階 | 実践ログ | 数字 |
